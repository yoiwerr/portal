# 提示词工程最佳实践

## Chain-of-Thought (CoT) 思维链

让模型在输出最终答案之前，先生成中间推理步骤。

### 适用场景
- 数学推理和逻辑问题
- 复杂决策分析
- 多步骤任务

### 示例
```
请逐步思考以下问题：
Step 1: 理解问题的核心目标
Step 2: 分析已知条件和约束
Step 3: 推导可能的解决方案
Step 4: 评估并选择最优方案
Step 5: 给出最终答案
```

### 注意事项
- 对于简单任务，CoT 可能过度复杂化
- 推理模型（如 DeepSeek-R1）天然支持 CoT
- 通用模型需要显式要求

## Few-Shot Prompting 少样本提示

在提示词中提供 2-5 个示例，帮助模型理解期望的输出格式和风格。

### 最佳实践
- 示例要覆盖不同的情况（边界 case）
- 示例格式必须与实际期望输出完全一致
- 示例数量 3-5 个最佳，过多会影响性能

## 角色扮演（Role Prompting）

为模型分配一个专业角色和背景，提升特定领域的输出质量。

### 角色设定要素
1. 专业领域和资历
2. 工作风格和方法论
3. 输出格式偏好
4. 约束和禁忌

### 示例
```
你是一个拥有10年经验的前端架构师，擅长 React 和 TypeScript。
你的代码风格：简洁、可维护、类型安全。
在给出建议时，优先考虑可维护性和团队协作效率。
避免推荐过新的实验性技术，除非明确标注风险。
```

## 结构化输出

要求模型以特定格式（JSON、表格、Markdown 等）输出，便于下游处理。

### 技巧
- 提供输出 schema 或模板
- JSON mode 只适用于支持的模型
- 对不支持 JSON mode 的模型，用 Markdown 代码块包裹

## 提示词优化的常见问题

1. **信息不足**: 提示词太模糊，模型无所适从
2. **过度约束**: 限制太多，模型无法发挥
3. **矛盾指令**: 提示词中存在互相冲突的要求
4. **缺少示例**: 对格式敏感的复杂任务缺少 Few-Shot

## 不同模型的提示策略

### 通用对话模型 (Qwen3, Claude Sonnet)
- 明确的角色设定 + 结构化输出约束
- 分步骤指令效果更好
- 需要更多上下文信息

### 推理模型 (DeepSeek-R1, Claude Opus)
- 利用其内置推理链
- 指令可以更抽象，让模型自行拆解
- 在 thinking 阶段给足自由度

### 代码模型 (DeepSeek-Coder, CodeLlama)
- 提供函数签名和 docstring
- 用注释说明期望行为
- 给出正反示例帮助理解
